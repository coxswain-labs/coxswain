# Conformance reports

One directory per Gateway API version Coxswain claims support for, named for the
**upstream** report directory rather than the CRD version — upstream keeps
per-patch directories through v1.4.x but unified minor directories from v1.5.
`.gateway-api-versions.json` maps one to the other.

```
conformance/reports/<report-dir>/coxswain-coxswain/standard-<impl-version>-default-report.yaml
```

Reports are produced by `scripts/run-conformance.sh --gateway-api-version vX.Y.Z`
against a cluster prepared by `scripts/setup-conformance.sh` with the same flag.
Gateway API CRDs are cluster-scoped singletons, so every version needs its own
fresh cluster.

A report is not a claim that every feature passed — it is a claim about the
feature set the cluster's CRDs can express. A run against Gateway API v1.4 has
fewer profiles and fewer `supportedFeatures` by design; see
`docs/src/reference/capability-matrix.md`.

Loose `*.yaml` directly under this directory is gitignored, so local iteration
does not dirty the tree.

## Upstream submission requirements

`kubernetes-sigs/gateway-api` requires each implementation folder to contain a
`README.md` with a table of contents (one row per report) and a `Reproduce`
section — a report set without them is not submittable. These are generated by
`scripts/render-conformance-readmes.sh`, which derives the table from the report
YAMLs so it cannot drift from what was actually run. Both
`scripts/run-conformance.sh` and the CI bundle job call it.

Upstream also requires `implementation.version` to be a **semver** matching a
release. A report produced off-tag gets `v0.5.0-40-g<sha>` from `git describe`,
which has no release page to link to — so **publishable reports must be produced
from a tagged commit**.

## Where reports are archived

**Not here, and not by CI.** Nothing in CI commits to this repository — the
durable archive is a GitHub release, whose assets never expire (Actions
artifacts are capped at 90 days on a public repo, which is a hard limit rather
than a setting).

Run the `Conformance reports (all Gateway API versions)` workflow with
`publish_release: true`. It attaches a tarball that unpacks to exactly this
layout, tagged with the same `git describe --long` string the reports carry
internally, and names the full source commit in the release body.

This directory is therefore a **manual staging area**: populate it (by
unpacking a release asset, or by running the scripts locally) when preparing a
publishing PR to `kubernetes-sigs/gateway-api`, which is where reports are
actually published.

Every report identifies its source tree — `implementation.version` and the
filename both carry a `git describe --tags --always --long` string, so the
commit ref is present even when the build sat on an exact release tag.
